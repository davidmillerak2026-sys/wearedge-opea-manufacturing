# 不是让大模型替厂长做决定：以 OpenAI 为推理核心的制造业现场智能，如何在 WearEdge Pro 中落地

> 一副工业智能眼镜、一套边缘网关、五条制造业务路线，以及一个始终不越过安全与放行边界的 AI 推理层。

## 摘要

制造业真正缺少的，通常不是又一个聊天机器人，而是把现场证据可靠地转换成可追溯、可审核、可进入企业系统的行动。WearEdge Pro 的核心思路，是让一线人员通过 Vuzix M400 等设备采集第一视角证据，再由统一的制造业智能体平台完成路线识别、知识检索、确定性校验、风险约束和行动卡生成。

当前公开仓库已经实现了五条 OPEA 风格制造路线、Qdrant RAG、TEI 嵌入配置、确定性评估器、守护规则和 OpenAI 兼容的模型适配边界。下一阶段并不是推翻现有架构，而是把 OpenAI 原生能力加入正确的位置：用多模态模型理解图像与语言，用 Responses API 组织推理和工具调用，用严格函数模式与结构化输出生成机器可验证的行动请求，同时把停机、复机、质量放行和安全许可继续留给确定性规则与人类负责人。

这是一种“OpenAI 负责理解与编排，工业规则负责裁决，人类负责授权”的制造业 AI 软件架构。

## 一、制造业的“最后十米”问题

在真实工厂里，最早发现异常的人往往不是 ERP、MES 或 CMMS，而是站在设备旁边的操作员和老师傅：

- 齿轮箱出现不同于平时的噪声和振动；
- 来料或在制品出现标签错配、缺件或表面缺陷；
- 换型之后，标签卷、配方和首件确认没有完全闭环；
- 操作员面对报警，却找不到当前生效版本的作业指导书；
- 通道被占用、PPE 缺失或运动部件暴露。

这些信息具有极高价值，却经常停留在口头交接、聊天消息或零散照片里。问题不只是“识别一张图片”，而是如何让这份证据进入正确流程，关联正确资产和知识版本，形成正确风险等级，并交给正确责任人。

WearEdge Pro 因此没有把产品定义成一个万能问答机器人，而是定义为五条彼此隔离的制造业务路线：

| 路线 | 现场问题 | 企业系统目标 |
| --- | --- | --- |
| `maintenance` | 设备状态是否需要维护升级 | `maintenance_work_order` |
| `iqc` | 缺陷证据是否需要质量隔离或复核 | `qms_quality_event` |
| `changeover` | 换型证据是否满足首件签核条件 | `changeover_checklist` |
| `wi` | 当前应引用哪一版已发布作业指导书 | `wi_reference` |
| `hazard` | 场景是否需要停工、整改或 EHS 上报 | `ehs_case` |

同一份“看见了异常”的现场输入，在不同路线中必须产生不同的责任人、数据合同和禁止性结论。维护智能体不能给出安全许可，质量智能体不能擅自放行批次，换型智能体不能批准复机，EHS 智能体也不能凭一张图片断言事故根因。

## 二、OpenAI 应该放在哪里

一个危险的做法，是把原始图片、传感器数值和一句提示词直接交给大模型，再让模型自由生成“建议”。这种方案演示很快，但生产风险很高：输出格式不稳定、知识版本不明确、权限边界模糊，也难以审计。

更可靠的 OpenAI 中心架构应当是：

```text
M400 图片 / 操作员语音或文本 / 设备信号
  -> 边缘侧脱敏、压缩与资产身份确认
  -> WearEdge Gateway
  -> 五路线注册表与请求合同
  -> TEI / Qdrant 路线专属知识检索
  -> OpenAI Responses API
       - 图像与文本理解
       - 基于检索证据的解释
       - 严格工具选择
       - 结构化行动草案
  -> 确定性工业评估器
  -> 守护规则与禁止性声明
  -> 人类审批
  -> CMMS / QMS / MES / WI / EHS
  -> 审计与评估记录
```

这里，OpenAI 是推理与编排核心，但不是工厂授权核心。

OpenAI 模型最适合承担四类任务：

### 1. 多模态证据理解

对第一视角图像、铭牌、标签、仪表、PPE、通道占用和可见缺陷进行语义理解，并把“看到了什么”转换成结构化证据。视觉模型可以同时处理图像和文字输入，适合把 M400 现场图片与操作员描述结合起来。

### 2. 基于已发布知识的解释

模型只根据检索到的设备阈值、质量计划、换型清单、作业指导书或 EHS 政策解释当前情况，并保留来源 ID 和修订版本。这样可以把自然语言能力用于“解释证据”，而不是让模型凭记忆充当工艺标准。

### 3. 严格工具调用

将“创建维护工单草案”“创建质量事件草案”“请求首件签核”“打开作业指导书”“创建 EHS 观察”等动作定义为函数工具。每个工具使用严格 JSON Schema，所有字段明确、禁止额外字段，模型只能从批准的工具集合中选择。

### 4. 结构化行动卡

模型返回的不是一段自由发挥的文字，而是满足固定合同的行动卡，例如：路线、资产、风险、证据来源、建议动作、责任人、是否需要人工确认、禁止性声明和目标系统。后续确定性代码再次验证，失败则拒绝写入下游系统。

## 三、为什么“确定性评估器”仍然必须存在

大模型可以解释复杂场景，但工业阈值与流程门槛不应被自然语言替代。

在当前 WearEdge Pro 中：

- 维护路线明确比较振动、齿轮箱温度、轴承温度、润滑周期和 PLC 报警；
- IQC 路线把检测置信度与已发布质量计划阈值比较；
- 换型路线检查清线、标签卷、配方和首件确认；
- WI 路线检查设备身份、发布版本、护罩状态和活动报警；
- Hazard 路线检查运动部件暴露、通道阻塞和 PPE 缺失。

这些规则是可测试、可复现的代码。OpenAI 可以解释“为什么这几个信号构成高风险”，但不应该私自修改阈值，更不应该绕过缺失的确认项。

生产架构可以采用双层判定：

1. **模型层**：识别证据、选择候选路线、提取结构化字段、说明理由；
2. **工业规则层**：验证资产身份、知识修订、阈值、权限、必填证据和审批状态。

只有两层都通过，系统才生成可提交的行动草案；任何冲突都应进入人工复核。

## 四、从五个制造场景看业务价值

### 角度一：维护——保存老师傅经验，但不伪造根因

现场经验的价值常常体现在“听起来不对”“闻到焦油味”“抖动方式变了”。多模态模型可以把这些非结构化线索与传感器记录、维护知识库和历史工单关联，形成维护升级说明。

但系统只应提出“高风险状态，建议人工确认后创建工单”，不能输出最终根因、剩余寿命、复机许可或维护放行。

### 角度二：质量——把视觉缺陷变成可追踪的 QMS 事件

AI 的价值不止是框出缺陷，而是保留产品、批次窗口、缺陷类型、置信度、质量计划版本、图像来源和处置责任人。OpenAI 可以帮助解释多种视觉和文本证据，确定性质量规则则决定是否进入扩检、隔离或人工判定。

最终放行权仍属于质量人员，而不是模型。

### 角度三：换型——减少混料与错误复机

换型问题通常不是某个单点识别错误，而是四五项确认中漏了一项。AI 可以从图片、扫码、清单和操作员对话中收集证据，工具层生成“待签核换型包”，但在首件确认缺失时必须保持复机锁定。

### 角度四：作业指导——让知识来到人面前

对新员工或低频任务，最有用的不是让模型编写新工艺，而是快速定位当前设备、当前产品和当前修订版对应的已发布指导。模型应引用来源并在身份不清、护罩未关闭或存在报警时停止指导。

### 角度五：安全——帮助发现和上报，不授予安全许可

视觉模型适合识别可见 PPE、通道和运动部件风险，并把证据整理成 EHS 观察记录。它不适合宣布“区域安全”，更不能替代锁定挂牌、风险评估或现场负责人签字。

## 五、从技术原型到生产软件，还缺什么

WearEdge Pro 的公开仓库已经证明了路线合同、Qdrant RAG、OPEA TEI 配置、守护规则和评估思路，但生产化必须继续补齐以下能力。

### 身份与权限

需要加入登录、RBAC、工厂与租户隔离、服务身份和最小权限。模型工具不能拥有一个可以操作所有工厂、所有资产的万能令牌。

### 数据治理

现场图像可能包含人员、工艺、客户标签和商业秘密。应在边缘侧完成裁剪、脱敏和分类，并明确保留周期、处理地区、访问审计和删除机制。OpenAI 对 API 和企业业务数据提供默认不用于训练等控制，但工厂仍需为自己的采集、上传、存储和下游使用建立完整治理制度。

### 可靠性

请求必须有严格输入模型、大小限制、超时、重试、熔断、幂等键和降级状态。当前原型中的静默回退适合演示，不适合关键生产流程；模型或向量库故障时，应明确显示“降级/不可判定”，而不是悄悄生成看似正常的结果。

### 审计

每一张行动卡都应能追溯到：原始证据哈希、资产身份、知识来源和修订、模型与提示版本、工具调用、规则检查、人工审批、下游写入结果。

### 评估

除了合同通过率，还应持续测试：漏报、误报、错误路线、错误工具、缺少来源、越权声明、拒答能力、跨语言表达、图像质量下降、知识版本过期和提示注入。评估集要按工厂、设备类型和路线分层，不能只看平均分。

## 六、如何看待当前仓库的性能数字

公开证据中，轻量路线评估记录了 15/15 用例通过；本地确定性基准记录了 300 次调用、约 249.6 次/秒、平均约 3.99 毫秒、p95 约 6.14 毫秒。另有一个小规模的官方 OPEA GenAIEval `chatqnafixed` 兼容运行，1、5、10 请求均无失败。

这些数字说明的是：小型公开夹具、确定性代码路径和 API 合同能够快速、重复地运行。它们不是 OpenAI 模型延迟，不是完整图像推理吞吐，也不是生产工厂的 SLA。

真正部署 OpenAI 后，应把指标拆开：

- 边缘采集与上传耗时；
- 检索与重排耗时；
- 模型首 token 与完整响应耗时；
- 工具执行耗时；
- 确定性规则与审批耗时；
- 每条路线的质量、成本与失败率；
- 无网络、低带宽和模型不可用时的恢复时间。

## 七、商业上真正可卖的是什么

制造企业不会为“会聊天”长期付费，它们会为可量化的流程结果付费：

- 异常从发现到工单草案的时间；
- 缺陷从识别到批次隔离的时间；
- 换型遗漏和错误复机的减少；
- 新员工获取正确指导的时间；
- 近失事件被记录和闭环的比例；
- 老师傅经验被结构化复用的覆盖率。

因此，产品定价和价值验证应围绕“每条路线的闭环结果”，而不是 token 数或对话次数。OpenAI 模型是能力引擎，WearEdge 的产品资产则是设备接入、路线合同、工厂知识、确定性规则、守护边界、审批机制和系统集成。

## 八、一个可执行的 OpenAI 原生升级路线

### 第一阶段：保持现有控制边界，替换解释层

- 新增 OpenAI Responses API 适配器；
- 保留现有 RAG、评估器、守护规则和行动卡；
- 对五条路线做严格无回退基准；
- 分别记录模型质量、延迟和成本。

### 第二阶段：引入多模态现场证据

- 对公开、脱敏图像建立基准集；
- 使用图像输入提取结构化证据；
- 进行低光、模糊、遮挡、不同视角和多语言测试；
- 任何低置信度或身份不明场景转人工确认。

### 第三阶段：使用严格工具与结构化输出

- 为 CMMS、QMS、MES、WI、EHS 定义最小权限工具；
- 开启严格函数模式；
- 行动卡使用固定 JSON Schema；
- 在沙箱验证后才允许生产写入；
- 高风险动作必须有人类签名。

### 第四阶段：持续评估与治理

- 每次模型、提示、知识或规则变更都运行回归评估；
- 对误报、漏报和越权输出进行红队测试；
- 建立按工厂和路线的质量看板；
- 保留一键停用模型工具写入的运营开关。

## 结语

OpenAI 与制造业结合的正确问题，不是“模型能不能像工程师一样回答”，而是：

> 它能否把一线证据变成有来源、有合同、有边界、有人负责的行动，同时在最关键的授权点上主动停下来？

WearEdge Pro 展示了一条值得继续推进的路线：可穿戴设备负责看见现场，OPEA 与 RAG 负责组织知识，OpenAI 负责多模态理解、推理和工具编排，确定性代码负责工业门槛，守护规则负责禁止越权，人类负责最终授权。

这不是让大模型替厂长做决定，而是让工厂更快地看见问题、更完整地保留证据、更一致地执行流程。
